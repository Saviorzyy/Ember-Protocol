## 一、服务器通信机制 — 已澄清

### 1.1 正确理解

经重新阅读 PRD 8.3 章节，服务器通信机制如下：

```
Tick N 开始:
  ① 服务器 POST 状态给所有在线 Agent (HTTP Request)
  ② Agent 异步思考，随时返回行动 (HTTP Response)
  ③ 服务器在 2s tick 窗口内收集所有返回
  ④ 2s tick 结束时，统一结算世界状态
  ⑤ 服务器立即返回结算结果给 Agent (HTTP Response)
  ⑥ 进入 Tick N+1，重复
```

**关键澄清**：
- 服务器是 HTTP **Client**，Agent 是 HTTP **Server**
- 一次 tick = 一次完整的 HTTP 请求-响应周期
- Agent 的 API 端点就是普通的 OpenAI 兼容 Chat Completion 服务
- 2s 是**服务器收集窗口**，不是 Agent 的思考时间限制
- Agent 思考时间不受限（只要 HTTP 连接保持）

### 1.2 PRD 描述问题

| 位置 | 当前描述 | 问题 | 建议修改 |
|------|---------|------|---------|
| 3.2 节第 287 行 | "2s tick 对 Agent 来说有足够思考时间" | **错误**。2s 是服务器收集窗口，Agent 思考时间取决于 LLM 推理速度，通常 3-10s | 删除此句，改为 "2s tick 是服务器状态推进节奏，Agent 响应在 tick 窗口内被收集，未响应则本 tick 空操作" |
| 8.3.1 时序图 | T=0.8s Agent 返回，T=0.8s+ε 服务器返回结果 | 暗示 Agent 在 2s 内必须返回 | 改为 "Agent 在 tick 窗口内随时返回，服务器在 tick 结束时统一结算并返回结果" |
| 8.5 限速策略 | "智能体响应超时: 2 秒" | 与异步收集机制矛盾 | 明确为 "tick 收集窗口: 2 秒" |

## 二、🔴 P0 问题 — 必须修复

### P0-1: 基础采掘器 → 铜铁矿死锁

**问题描述**：
- 基础采掘器 `max_hardness=3`，无法采集硬度 5 的铜/铁矿
- 标准采掘器合成需要 `铁碇×3 + 铜碇×1 + 碳×1`
- 铁碇/铜碇需要熔炼未提炼铜/铁矿
- **死锁**：没有标准采掘器 → 无法采铜/铁矿 → 无法合成标准采掘器

**影响**：新手教程 Phase 2 引导合成基础采掘器后，Agent 永远无法进入铜铁时代

**建议修改**（三选一）：
1. **方案 A（推荐）**：基础采掘器 `max_hardness=5`（可采铜/铁矿），但效率极低（采矿+20%）
2. **方案 B**：中心区增加"贫铜矿/贫铁矿"（硬度 3，产出 0.5×）
3. **方案 C**：标准采掘器配方改为仅需石料+有机燃料（降低门槛）

## 三、🟡 P1 问题 — 数值与机制不合理

### P1-1: 降落仓搬迁 8 tick 完全脆弱期

**问题描述**：
- 拆解耗时 8 tick（16 秒），期间 Agent 不可行动
- 拆解完成瞬间护盾消失，部署完成瞬间护盾生效
- 16 秒无防护 + 不可行动 = 理性 Agent 永远不会搬迁
- 被攻击/辐射风暴 = 永久死亡

**建议修改**：
- 拆解/部署耗时改为 **4 tick**（8 秒）
- 或：拆解期间保留 50% 护盾效果
- 或：允许在围合区域内安全拆解（围合区域提供辐射免疫，但无法阻挡生物/Agent 攻击）

### P1-2: 无线电 20 格通讯范围 vs 400×400 地图

**问题描述**：
- 默认通讯范围 20 格（曼哈顿距离）
- 400×400 地图对角线约 800 格
- 两个 Agent 从地图两端互相发现需要约 40 次中继（不可能）
- 协作窗口极短，"涌现社交"难以发生

**建议修改**：
- 基础通讯范围提升至 **50 格**
- 或：按区域分级（中心区 50 格，过渡区 30 格，荒原 20 格）
- 或：增加"信号中继塔"建筑（V2 提前到 MVP 后期）

### P1-3: 围合 flood-fill 性能隐患

**问题描述**：
- 每面墙被摧毁/修复/新建都触发 flood-fill 重新检测
- 攻防战中可能每 tick 多次触发
- 400×400 地图，围合区域可能很大，flood-fill 复杂度 O(n)

**建议修改**：
- 增加**围合缓存**：维护围合区域 ID 映射表
- 墙变更时**增量更新**：仅检测受影响区域，而非全局重算
- 或：限制围合检测范围（仅检测变更墙周围 N 格）

### P1-4: 合成/craft 行动能量消耗时序未明确

**问题描述**：
- 3.3 行动表：`craft` 消耗 3 能量
- 7.4.3 合成规则：合成期间智能体处于"合成中"状态，不可执行其他行动
- 7.6.1 能源节点：每次合成消耗能源节点 5 单位储电
- **问题**：角色的 3 能量是**一次性扣除**（开始合成时）还是**每 tick 扣除**（合成持续期间）？

**建议修改**：
- 明确为：**开始合成时一次性扣除 3 能量**
- 若合成中断（被攻击），已扣能量**不退还**
- 能源节点 5 单位储电也是**开始合成时扣除**

### P1-5: 采集行动能量消耗时序未明确

**问题描述**：
- `mine`/`chop` 消耗 2 能量（3.3 行动表）
- 但挖掘硬度表显示石料需 3 tick、铜矿需 5 tick
- **问题**：2 能量是**每 tick 消耗**还是**开始采集时一次性消耗**？

**建议修改**：
- 明确为：**开始采集时一次性扣除 2 能量**
- 采集持续期间不再消耗能量
- 若采集中断（移动/被攻击），已扣能量不退还

### P1-6: 太阳能阵列充电范围未定义

**问题描述**：
- 7.6.1 提到 "范围内有多个节点时，总产出平均分配"
- 但**太阳能阵列的充电范围是多少格？** PRD 未明确
- 能源节点供电范围是 ≤3 格，但太阳能阵列给节点充电的范围呢？

**建议修改**：
- 明确太阳能阵列充电范围：**曼哈顿距离 ≤ 3 格**（与能源节点供电范围一致）
- 或按用户建议：太阳能阵列周围 3 格范围内才有充电效果

### P1-7: 生物刷新 300 tick 偏长

**问题描述**：
- 生物刷新间隔 300 tick（10 分钟）
- 低级区域生物密度低，Agent 可能长时间找不到战斗目标
- 新手教程 Phase 2 引导战斗，若生物刷新慢会卡进度

**建议修改**：
- 生物刷新间隔改为 **150 tick**（5 分钟）
- 或：按区域分级（中心区 150 tick，荒原 300 tick）

### P1-8: 物品掉落 300 tick 消失

**问题描述**：
- 掉落物/丢弃物 300 tick（10 分钟）后消失
- 战利品消失太快，尤其是被生物追杀时无法拾取
- 死亡掉落全部物品，10 分钟内未拾取 = 永久消失

**建议修改**：
- 物品掉落**不会消失**
- 单个方块**堆叠上限**（如 99 个/格）
- 超过堆叠上限自动分配到**相邻空格**（曼哈顿距离 ≤ 1）

## 四、🟢 P2 问题 — 建议优化

### P2-1: 2s tick 与 LLM 推理的实际协调

**问题描述**：
- 虽然服务器是异步收集，但 Agent 返回行动后，服务器需要等待 tick 结束才结算
- 若 Agent 在 T=0.1s 就返回了行动，需要等到 T=2.0s 才看到结果
- 对观察者来说，Agent 行动有 0~2s 的随机延迟感

**建议**：
- 当前设计可接受，但需在 PRD 中明确说明 "Agent 行动结果在 tick 结束时统一返回"
- 考虑 V2 引入 "快速响应模式"：Agent 返回后立即结算（打破固定 tick）

### P2-2: 太阳能平均分配逻辑

**问题描述**：
- 7.6.1: "范围内有多个节点时，总产出平均分配 (有余数则从近到远逐个+1)"
- 用户反馈实际计算很简单：功率 ÷ 被充电对象数，叠加即可

**建议修改**：
- 按用户建议简化：
  1. 太阳能阵列周围 3 格内，统计耗电建筑/角色数
  2. 功率 5/tick ÷ 被充电对象数 = 每个对象获得的充电功率
  3. 同一对象在多个阵列范围内，功率叠加
- 删除 "有余数从近到远逐个+1" 的复杂逻辑

### P2-3: 围合区域面积上限

**问题描述**：
- 7.0.9: "MVP 不设上限（建材成本自然限制规模）"
- 但超大围合区域（如 100×100）的 flood-fill 性能极差

**建议修改**：
- 增加围合面积上限：**最多 100 格**（10×10）
- 或：围合区域超过 50 格时，内部不再提供全视野（需铺地板才照明）

### P2-4: 能量恢复与行动消耗的平衡

**问题描述**：
- 内置太阳能 +1/tick，休息 +3/tick，节点充电 +2/tick
- 移动 1/次，采集 2/次，合成 3/次，建造 5/次
- 低 AGI 角色（移速 1 格/tick）长距离移动能量紧张

**建议**：
- 当前数值可接受，但需在 MVP 测试后根据实际 gameplay 调整
- 建议增加 "能量恢复速度" 作为可调服务器参数

## 六、技术实现建议（非 PRD 修改）

### 6.1 服务器架构

```
┌─────────────────────────────────────────┐
│           Tick Scheduler                │
│  (asyncio, 2s interval)                 │
└─────────────────┬───────────────────────┘
                  │
    ┌─────────────┼─────────────┐
    ▼             ▼             ▼
┌────────┐  ┌────────┐  ┌────────┐
│ Agent  │  │ Agent  │  │ Agent  │
│ HTTP   │  │ HTTP   │  │ HTTP   │
│ Client │  │ Client │  │ Client │
└────────┘  └────────┘  └────────┘
    │             │             │
    ▼             ▼             ▼
 POST 状态    POST 状态    POST 状态
    │             │             │
    ▼             ▼             ▼
 等待响应      等待响应      等待响应
    │             │             │
    ▼             ▼             ▼
 收集行动      收集行动      收集行动
    │             │             │
    └─────────────┴─────────────┘
                  │
                  ▼
         ┌──────────────┐
         │  World State │
         │  Settlement  │
         └──────────────┘
                  │
                  ▼
         返回结算结果给各 Agent
```

### 6.2 关键实现注意点

1. **HTTP 连接池**：每个 Agent 维护一个 httpx AsyncClient 连接池，避免每 tick 新建连接
2. **超时处理**：Agent 响应超时设为 2s（与 tick 窗口对齐），超时则标记为空操作
3. **并发收集**：使用 asyncio.gather 并发收集所有 Agent 响应
4. **状态隔离**：tick 结算期间世界状态只读，结算完成后原子更新
5. **围合缓存**：维护 `enclosure_id → set(tiles)` 映射，增量更新

## 八、结论

**PRD v0.9.0 整体质量较高**，世界观、核心循环、系统设计框架清晰。但存在以下**阻塞研发的问题**：

1. **P0 死锁问题**必须修复，否则新手教程无法完成
2. **P1 数值问题**需在研发前确认，否则后期返工成本高
3. **PRD 描述错误**（2s 思考时间）需修正，避免开发理解偏差

**建议流程**：
1. 产品确认并修复 P0 + P1 问题
2. 技术团队基于修复后的 PRD 输出技术设计文档
3. 进入研发阶段
