# Ember Protocol — MVP PRD vs 代码实现差异分析报告

> **分析日期**: 2026-05-08  
> **PRD 版本**: v1.3.2-mvp  
> **分析范围**: `server/`、`web/src/` 全部源代码 vs `docs/PRD-MVP.zh-CN.md`  
> **修订说明**: v1.3.1 统一了 Rock/Stone 层叠设计描述——Stone（L2）覆盖在 Rock（L1 基岩）之上，阻挡通行，站在旁边开采，采尽后露出 Rock 形成矿道（2D Minecraft 式洞穴挖掘）。本报告已据此修正相关条目。

## 二、逐模块差异清单

### 2.1 地形与世界生成 (PRD §7.0, §7.13)

> **设计核心**：Stone（L2）覆盖在 Rock（L1 基岩）之上，阻挡通行（类似 Minecraft 石头方块），Agent 站在相邻非 Stone 格上开采。采尽后 Stone 消失，露出下方 L1 基岩形成矿道。

| # | 差异项 | PRD 要求 | 代码现状 | 严重度 |
|---|--------|---------|---------|--------|
| T-1 | **Stone 阻挡通行** | Stone 作为 L2 覆盖物阻挡移动，Agent 需站在相邻格开采 | `Tile.passable` 中 `stone_amount > 0` 返回 False，**正确实现** | ✅ |
| T-2 | **站在相邻格开采** | Agent 必须站在非水/非沟壑/非Stone格上开采相邻 Stone | `_do_mine()` 检查 `dist > 1` + Agent 站立格不是 Stone，**正确实现** | ✅ |
| T-3 | **Stone 覆盖在 Rock 上** | 地形生成时 L1 Rock 格覆盖 L2 Stone，采尽后暴露 Rock 矿道 | `terrain_gen.py` 第 65-68 行：Rock 格上分配 Stone，**正确实现** | ✅ |
| T-4 | **采尽后 L1 保持不变** | Stone 采尽后 L2 消失，露出地形生成时的原始 L1（多为 Rock） | `world.py:651-653`：采尽后 `tile.l1 = Terrain.ROCK if random.random() < 0.35 else Terrain.FLAT`——**随机重分配 L1，而非保留原始值**。导致 Stone 下方原本是 Rock 的基岩可能变成 Flat | 🟡 中 |
| T-5 | **`can_have_l2_stone` 限制** | PRD v1.3.1: Rock 可拥有 L2 Stone | `models.py:186-188` 的 `can_have_l2_stone` 仅允许 flat/sand——**与 PRD 不符**，但此属性当前未被运行时逻辑调用（仅地形生成时直接赋值），影响有限 | 🟢 低 |
| T-6 | **柏林噪声参数** | PRD §7.13.2: 3 层 octave, scale 0.03/0.08/0.20 | `terrain_gen.py` 使用 fbm(0.04, 3), fbm(0.18, 2), fbm(0.05, 2)——参数不完全一致但效果近似 | 🟢 低 |

## 四、按严重度汇总

### 🔴 高严重度（阻塞 MVP 核心玩法）

1. **生物系统完全缺失** — 无生成、无 AI、无掉落、Agent 无法攻击生物
2. **辐射伤害未实现** — 区域辐射和风暴伤害均未实现，生存压力为零
3. **Power Node 无线充电未实现** — 能源节点的核心功能缺失
4. **护盾攻击免疫未实现** — 外界可攻击护盾内 Agent，新手无安全区
5. **WAL 日志未实现** — 服务器崩溃丢失全部世界状态
6. **崩溃恢复未实现** — 启动时不加载快照
7. **Stone 采尽后 L1 随机重分配** — 应保留地形生成时的原始 L1（多为 Rock），当前代码随机分配可能破坏矿道基岩地板
8. **耐久度追踪 bug** — 装备后物品无法正确追踪耐久度
