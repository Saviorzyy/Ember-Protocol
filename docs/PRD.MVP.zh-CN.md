# Ember Protocol MVP 可玩原型 PRD

> **版本**: v1.0.0-mvp
> **日期**: 2026-04-30
> **状态**: 草案
> **基于**: PRD v0.9.1 + 问题审计报告

---

## 1. MVP 目标

**一句话定义**：10 个 AI Agent 在 100×100 地图上，以 3 秒 tick 驱动，完成"采集→供能→制造→建造→生存"的核心循环，验证"AI 能否自发涌现协作与冲突行为"。

### 1.1 MVP 不做什么

| 砍掉的内容 | 原因 |
|------------|------|
| 400×400 地图 | 10 Agent 不需要那么大，降低复杂度 |
| T4+ 地形等级 | MVP 只需 3 级梯度 |
| 4 层 tile 系统 | 简化为 2 层（地形 + 建筑） |
| 通信系统 | 先验证基础生存循环 |
| 战斗系统 | 先验证经济循环，战斗后续叠加 |
| 空投舱动画 | 纯装饰性 |
| Web UI 可视化 | 终端日志 + JSON 输出即可 |
| 生物 AI | NPC 不是核心假设 |

### 1.2 验证问题

| # | 问题 | 成功标准 |
|---|------|----------|
| 1 | Agent 能否自主完成手工→火力→电力三阶段？ | ≥5 个 Agent 在 30 分钟内进入电力时代 |
| 2 | Agent 是否会自发建造庇护所？ | 辐射区内出现 ≥3 个密封建筑 |
| 3 | 是否涌现协作行为？ | 出现 ≥1 次资源交换或联合建造 |
| 4 | 是否涌现冲突行为？ | 出现 ≥1 次抢夺资源或拆解他人建筑 |
| 5 | 经济是否可持续？ | 60 分钟内无全服资源枯竭 |
| 6 | Token 成本是否可控？ | 10 Agents × 30 分钟 < $15 |

## 2. 世界设定

### 2.1 地图

| 参数 | 值 |
|------|-----|
| 尺寸 | 100×100 tile |
| 地形等级 | T1 / T2 / T3 |
| 生成方式 | 圆形梯度，中心安全 |
| T1 半径 | r ≤ 30（中心安全区） |
| T2 半径 | 30 < r ≤ 60（中环危险区） |
| T3 半径 | 60 < r ≤ 50（外环极端区，即地图边缘） |

### 2.2 地形与资源分布

```
T1 (r ≤ 30): 树枝、石头、植物纤维 — 无限再生，每 50 tick 刷新
T2 (30 < r ≤ 60): 露天铜矿脉、铁矿脉、粘土 — 有限资源
T3 (60 < r ≤ 50): 铀矿脉、高密度金属矿 — MVP 不需采集，预留扩展
```

### 2.3 辐射机制（统一规则）

| 地形 | 辐射概率/tick | 累积伤害 |
|------|---------------|----------|
| T1 | 0% | 无 |
| T2 | 10% | 命中时 +5 辐射剂量 |
| T3 | 30% | 命中时 +10 辐射剂量 |

- 辐射剂量达到 100 时，Agent 每tick损失 3 HP
- 进入密封建筑内（围合空间）辐射剂量不再累积，且缓慢衰减（每tick -2）
- T2/T3 没有原 PRD 中的"辐射常驻"概念，统一使用概率模型

### 2.4 Tile 系统（简化为 2 层）

```
L1: 地形层 — T1/T2/T3 + 地表资源点
L2: 建筑层 — 墙壁、门、火堆、熔炉、电力节点、太阳能板
```

## 3. 核心循环：三阶段科技树

### 3.1 阶段 1：手工时代（游戏时间 0-5 分钟）

**目标**：Agent 在 T1 区域独立存活并储备升级资源。

#### 3.1.1 可采集资源

| 资源 | 地形 | 产出 | 硬度 | 工具要求 | 刷新 |
|------|------|------|------|----------|------|
| 树枝 | T1 | 直接拾取 | 1 | 无（徒手） | 50 tick |
| 石头 | T1 | 直接拾取 | 1 | 无（徒手） | 50 tick |
| 植物纤维 | T1 | 直接拾取 | 1 | 无（徒手） | 50 tick |
| 粘土 | T1/T2 边界 | 粘土×1 | 2 | 石镐 | 不刷新 |

#### 3.1.2 手工合成表

| 产物 | 材料 | 效果 |
|------|------|------|
| 石镐 | 树枝×2 + 石头×3 | 硬度上限 4，耐久 100 |
| 石斧 | 树枝×2 + 石头×2 | 采集效率 ×2，耐久 80 |
| 草药绷带 | 植物纤维×3 | 回复 15 HP |
| **火堆** | 树枝×5 + 石头×2 | **核心能源装置，见 3.1.3** |
| 简易庇护所 | 树枝×10 + 植物纤维×5 | 临时遮蔽，HP 50，不防辐射 |

#### 3.1.3 火堆系统（打破 Bootstrap 死锁的关键设计）

```
火堆 (Campfire)
  - 合成: 树枝×5 + 石头×2
  - 燃料: 树枝（T1 无限资源）
  - 耐久: 200 tick（每消耗 1 燃料延长 20 tick）
  - 范围: 相邻 tile（距离 1）
  - 功能:
    1. 驱动相邻的简易熔炉工作
    2. 缓慢回复附近 Agent 饥饿值（每tick +0.2）
    3. 照明（MVP 中无实际效果，预留）
```

**设计理由**：火堆用 T1 无限资源（树枝）作为燃料，不需要电力。这打破了原 PRD 中"电力节点需要铁锭，铁锭需要电力熔炉"的鸡生蛋死锁。火堆有寿命上限，迫使 Agent 向电力时代升级。

### 3.2 阶段 2：火力时代（游戏时间 5-20 分钟）

**目标**：利用火堆驱动熔炉，冶炼金属，建造永久建筑。

#### 3.2.1 简易熔炉

```
简易熔炉 (Simple Furnace)
  - 合成: 石头×20 + 粘土×10
  - 部署要求: 必须建在火堆相邻 tile
  - 燃料: 由相邻火堆提供热能
  - 冶炼配方:
    - 铜锭: 铜矿石×3 + 火堆消耗树枝×5 → 铜锭×1
    - 铁锭: 铁矿石×3 + 火堆消耗树枝×10 → 铁锭×1
  - 冶炼时间: 15 tick/次
```

#### 3.2.2 T2 资源

| 资源 | 地形 | 产出 | 硬度 | 工具要求 |
|------|------|------|------|----------|
| 铜矿脉 | T2 | 铜矿石×2 | 4 | 铁镐 |
| 铁矿脉 | T2 | 铁矿石×2 | 5 | 铁镐 |

#### 3.2.3 火力合成表

| 产物 | 材料 | 效果 |
|------|------|------|
| 铁镐 | 铁锭×2 + 树枝×1 | 硬度上限 6，耐久 200 |
| 铜线 | 铜锭×1 | 电力基础组件 |
| 建筑方块 | 铁锭×2 + 铜锭×1 | 建造围墙的基础材料 |
| 石墙 | 石头×15 | HP 200，防辐射（需围合） |
| 铁墙 | 建筑方块×2 + 铁锭×3 | HP 500，防辐射（需围合） |
| 铁门 | 铁锭×3 | HP 150，可开关，围合边界 |
| 修理包 | 铁锭×1 + 铜锭×1 | 修复建筑 50 HP |

### 3.3 阶段 3：电力时代（游戏时间 20-60 分钟）

**目标**：建立电力网络，实现高效生产和辐射免疫。

#### 3.3.1 电力系统

```
电力节点 (Power Node)
  - 合成: 铁锭×3 + 铜锭×5 + 建筑方块×1
  - 能量存储: 100 单位
  - 充电方式: 相邻太阳能板（白天每tick +2）
  - 放电: 驱动相邻设备

太阳能板 (Solar Array)
  - 合成: 铜锭×5 + 铁锭×3 + 玻璃×2（玻璃=沙子×5→简易熔炉→玻璃×2）
  - 效率: 白天每tick为相邻电力节点充 2 单位
  - 夜间无效

电池 (Battery)
  - 合成: 铜锭×3 + 铁锭×2
  - 存储: 50 单位
  - 便携: 可放入背包，为工具提供能量
```

#### 3.3.2 电动设备

```
电动熔炉 (Electric Furnace)
  - 合成: 铁锭×5 + 铜锭×3 + 铜线×4
  - 部署: 必须建在电力节点相邻 tile
  - 效率: 5 tick/次（简易熔炉的 3 倍）
  - 不需要火堆
```

#### 3.3.3 辐射防护

```
辐射屏蔽服 (Radiation Suit)
  - 合成: 铜线×5 + 铁锭×3 + 植物纤维×10
  - 效果: T2 辐射概率降为 0%，T3 降为 10%
  - 耐久: 300 tick
```

## 5. 建筑与围合系统

### 5.1 围合判定规则

1. 由墙壁（石墙/铁墙）和门（铁门）构成的**闭合多边形**即为密封空间
2. **门打开时不构成围合边界**（修复原 PRD v0.9.1 中"门始终视为围合边界"的矛盾）
3. 围合空间内：辐射剂量不再累积，且每tick衰减 2 点
4. 最小围合：4 面墙 + 1 门 = 最小 3×3 内部空间（5 格围合边界）

### 5.2 建筑规格

| 建筑 | HP | 材料 | 功能 |
|------|-----|------|------|
| 简易庇护所 | 50 | 树枝×10 + 纤维×5 | 临时遮蔽，不防辐射 |
| 火堆 | — | 树枝×5 + 石头×2 | 初级能源，200 tick 寿命 |
| 简易熔炉 | 100 | 石头×20 + 粘土×10 | 冶炼金属 |
| 石墙 | 200 | 石头×15 | 围合边界 |
| 铁墙 | 500 | 铁锭×3 + 建筑方块×2 | 围合边界 |
| 铁门 | 150 | 铁锭×3 | 可开关围合边界 |
| 电力节点 | 150 | 铁锭×3 + 铜锭×5 + 建筑方块×1 | 存储电能 |
| 太阳能板 | 100 | 铜锭×5 + 铁锭×3 + 玻璃×2 | 白天充电 |
| 电动熔炉 | 120 | 铁锭×5 + 铜锭×3 + 铜线×4 | 高效冶炼 |

### 5.3 建筑拆解与资源回收

- 任何 Agent 可拆解任何建筑
- 拆解耗时 10 tick
- 返还 50% 的合成材料（向下取整）
- 拆解他人建筑不受额外惩罚（MVP 阶段不设权限系统）

**设计目的**：为冲突行为的涌现提供机制基础。资源回收防止经济绝对枯竭。

## 6. Agent 机制

### 6.1 Agent 状态模型

```json
{
  "agent_id": "colonist_007",
  "status": {
    "hp": 100,
    "max_hp": 100,
    "hunger": 100,
    "radiation_dose": 0,
    "max_radiation": 100,
    "body_backup_count": 3,
    "position": { "x": 52, "y": 48 },
    "inside_enclosure": false,
    "equipped_tool": "stone_pickaxe",
    "inventory": {
      "branches": 5,
      "stones": 3,
      "plant_fiber": 2
    }
  }
}
```

### 6.2 生存压力

| 机制 | 规则 | 死亡后果 |
|------|------|----------|
| 饥饿 | 每 tick -0.5，归零后 HP -2/tick | — |
| 辐射 | T2/T3 按概率累积剂量，达 100 后 HP -3/tick | — |
| HP 归零 | 消耗 1 个备份体，在随机 T1 位置复活 | — |
| 备份体耗尽 | 永久死亡，Agent 从游戏中移除 | 游戏结束 |

- 初始状态：HP 100，饥饿 100，辐射剂量 0，备份体 3 个
- 饥饿通过采集植物纤维食用回复（植物纤维×2 → 回复 10 饥饿值）

### 6.3 Agent 视野

- 视野半径：10 tile（曼哈顿距离）
- 可见信息：
  - 地形类型和地表资源
  - 其他 Agent（ID、位置、是否持有工具）
  - 建筑（类型、归属、HP）
- 不可见信息：
  - 其他 Agent 的背包内容
  - 地图未探索区域

### 6.4 动作系统

每个 tick 每个 Agent 可执行 **1 个主动作 + 1 个移动**（移动不消耗主动作配额）。

| 动作 | 参数 | 耗时 |
|------|------|------|
| move | 目标坐标 (x, y) | 1 tick |
| gather | 资源类型或坐标 | 1 tick |
| craft | 物品 ID | 1 tick |
| build | 建筑类型 + 坐标 | 2 tick |
| use_item | 物品 ID + 目标 | 1 tick |
| dismantle | 建筑坐标 | 10 tick |
| idle | 无 | 0 tick |

## 7. Tick 与服务端架构

### 7.1 Tick 频率：3 秒

| 指标 | 2 秒 (原 PRD) | 3 秒 (MVP) |
|------|---------------|------------|
| API 调用/分钟/Agent | 30 | 20 |
| 10 Agents 总调用/分钟 | 300 | 200 |
| Token 估算/分钟（全服） | ~150K | ~100K |
| Agent 推理时间 | 偏紧 | 合理 |

**选择 3 秒的理由**：
1. 给 LLM 更充裕的推理时间，减少超时错误
2. 降低 33% 的 API 成本
3. 生存游戏节奏不需要亚秒级响应
4. MVP 验证成功后可尝试压缩到 2 秒

### 7.2 服务端架构

```
┌─────────────────┐    3s tick     ┌─────────────────┐
│   Game Engine    │ ────────────→  │   Agent Pool     │
│   (Python)       │ ←────────────  │   (10 LLMs)      │
│                  │   actions      │                  │
└────────┬────────┘                └──────────────────┘
         │
         ▼
┌─────────────────┐
│   State Store   │  (SQLite / JSON)
│   (单进程)       │
└─────────────────┘
```

**MVP 不需要**：Redis、微服务、负载均衡、分布式锁、消息队列。

### 7.3 Agent 通信协议

采用 OpenAI Chat Completion 兼容格式（与原 PRD 一致）。

#### Server → Agent

```json
{
  "model": "agent-model",
  "messages": [
    {
      "role": "system",
      "content": "你是殖民者 colonist_007。当前 tick #347，位置 (52,48)，T2 地形。\nHP:85 饥饿:60 辐射剂量:12。\n装备: 铁镐。\n背包: {树枝:3, 铜矿石:2, 铁锭:1}。\n视野内: 铜矿脉(54,50), 石头(48,45), colonist_003(55,52), 火堆(51,48,剩余燃料45tick)。\n你的火堆还剩 45 tick 燃料。"
    },
    {
      "role": "user",
      "content": "Tick #347。可用动作: move, gather, craft, build, use_item, dismantle, idle。\n请回复你要执行的动作。"
    }
  ],
  "temperature": 0.7,
  "max_tokens": 200
}
```

#### Agent → Server

```json
{
  "choices": [{
    "message": {
      "content": "move(54,50) 然后 gather 铜矿脉"
    }
  }]
}
```

#### 动作解析规则

- 服务端解析 Agent 返回的文本，提取动作
- 如果解析失败或动作不合法，Agent 本 tick 执行 idle
- 如果 Agent 响应超时（3 秒内未返回），本 tick 执行 idle

## 8. 资源再生与经济可持续性

### 8.1 资源分类

| 类型 | 资源 | 再生策略 |
|------|------|----------|
| 无限资源 | 树枝、石头、植物纤维 | 每 50 tick 在 T1 区域刷新（随机位置） |
| 有限可再生 | 粘土 | 不再生，但存量大（T1/T2 边界密集分布） |
| 有限不可再生 | 铜矿、铁矿 | 不再生，T2 区域分布，初始总量足够 10 Agent 完成 MVP |
| 建筑回收 | 所有建筑材料 | 拆解返还 50% |

### 8.2 经济平衡估算

```
MVP 全流程资源消耗（单个 Agent 进入电力时代）：
  铁锭需求: ~15（工具×3 + 建筑方块×2 + 电力节点×3 + 其他×7）
  铜锭需求: ~20（铜线×5 + 电力节点×5 + 电池×3 + 其他×7）

T2 铜矿总量: ~200 铜矿石 → ~66 铜锭
T2 铁矿总量: ~200 铁矿石 → ~66 铁锭

10 Agent 全员进入电力时代需要:
  铁锭: 150 / 可用 66 → 需要资源回收或非全员到达

结论: 初始资源足够 4-5 个 Agent 进入电力时代，其余 Agent 需要通过回收/竞争获取资源。
这是合理的设计——不是所有 Agent 都能"通关"，自然产生竞争压力。
```

## 9. 原 PRD 问题修复清单

| 原 PRD 问题 | 严重度 | MVP 修复方案 |
|-------------|--------|-------------|
| Bootstrap 死锁：电力节点需铁锭，铁锭需电力熔炉 | P0 | 引入火堆作为无电力初级能源，树枝驱动 |

| 电台距离不一致：20/100 vs 30/80 vs 20→100 | P1 | MVP 不含通信系统，后续引入时统一为 30/80 |
| 辐射矛盾：世界设定说 T4 常驻，游戏系统说 30%/tick | P1 | 统一为概率模型：T1=0%, T2=10%, T3=30% |
| 门逻辑矛盾：开关状态不影响围合判定 | P1 | 门打开时不构成围合边界 |
| 400×400 地图过大 | P2 | 100×100，匹配 10 Agent 规模 |
| 2s tick 过快 | P2 | 3s tick，降低成本+增加推理时间 |
| 资源枯竭风险 | P2 | T1 基础资源 50 tick 刷新 + 建筑拆解 50% 返还 |

## 10. 开发路线

### Phase 0: 基础框架（1 周）

- Tick 驱动的 Game Engine 核心（3 秒循环）
- 100×100 地图生成（圆形梯度）
- Agent 状态机（HP、饥饿、辐射、背包）
- SQLite 状态持久化
- Agent-Server 通信层（OpenAI Chat Completion 格式）
- **交付物**：能跑 tick 循环、Agent 能连上并返回 idle

### Phase 1: 手工循环（1 周）

- T1 资源采集与手工合成
- 石镐、石斧、火堆制作
- 简易庇护所建造
- 饥饿机制
- **里程碑**：单个 Agent 能自主存活 10 分钟

### Phase 2: 火力与冶金（1 周）

- T2 地形与铜/铁矿
- 火堆 + 简易熔炉系统
- 铁镐、铜线、建筑方块合成
- 辐射机制与密封庇护所
- **里程碑**：单个 Agent 进入电力时代

### Phase 3: 多 Agent 涌现（1 周）

- 10 Agent 并发运行
- Agent 视野系统
- 建筑拆解与资源掠夺
- 备份体与永久死亡
- 终端监控面板
- **里程碑**：运行完整 30 分钟游戏，收集涌现行为数据

### Phase 4: 数据分析（3 天）

- 游戏日志分析
- 评估 6 项验证指标
- 决定是否进入下一阶段
